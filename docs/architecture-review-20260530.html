Collapse try/catch into an async route wrapper

Strong in-process

auth.routes.ts, listings.routes.ts, orders.routes.ts, new shared/middleware/async-handler.ts

Before — 3 route files × N handlers

                flowchart TD
                  R1[auth/register] --> T1[try/catch/next]
                  R2[auth/login] --> T2[try/catch/next]
                  R3[auth/refresh] --> T3[try/catch/next]
                  R4[auth/me] --> T4[try/catch/next]
                  L1[listings/* x6] --> T5[try/catch/next]
                  O1[orders/* x6] --> T6[try/catch/next]
                  T1 --> S1[authService]
                  T2 --> S2[authService]
                  T3 --> S3[authService]
                  T4 --> S4[authService]
                  T5 --> LS[listingsService]
                  T6 --> OS[ordersService]
                  classDef boilerplate fill:#fef2f2,stroke:#dc2626,stroke-width:1px;
                  class T1,T2,T3,T4,T5,T6 boilerplate

Each handler independently replicates the error-forwarding contract.

After — wrapper absorbs the boilerplate

asyncHandler

catches rejected promises, forwards to next(err)

handler → call service
handler → call service
handler → call service

One interface. Route handlers declare intent; wrapper handles the contract.

Problem

Every route handler wraps its body in identical try { ... } catch (err) { next(err) }. The interface (what a handler must do) is nearly as complex as the implementation (the boilerplate). No locality.

Solution

An asyncHandler(fn) wrapper that catches rejected promises and forwards to next. Express 5 ships this natively; for Express 4, it's 4 lines.

Wins

  • Delete ~25 lines of boilerplate
  • Leverage: one wrapper, 13+ call sites
  • Locality: error forwarding lives in one place
  • Interface shrinks: handlers declare intent, not plumbing

Extract pagination into a shared module

Strong in-process

listings.service.ts (list, getBySeller), orders.service.ts (listBuyerOrders, listSellerOrders), new shared/pagination.ts

Before — 4 independent copies of the same pattern

listings.list()
offset = (page - 1) * limit
Promise.all([query, count])
{ data, pagination }
listings.getBySeller()
offset = (page - 1) * limit
Promise.all([query, count])
{ data, pagination }
orders.listBuyerOrders()
offset = (page - 1) * limit
Promise.all([query, count])
{ data, pagination }
orders.listSellerOrders()
offset = (page - 1) * limit
Promise.all([query, count])
{ data, pagination }

Contract change requires 4 edits. No locality.

After — one deep module, 4 thin callers

paginate

(query, countQuery, page, limit) → PaginatedResult

list() → paginate(builtQuery, countQuery, page, limit)
getBySeller() → paginate(...)
listBuyerOrders() → paginate(...)
listSellerOrders() → paginate(...)

One interface. 4 callers delegate. Contract change = 1 edit.

Problem

The pagination pattern — offset calculation, parallel data+count queries, { data, pagination } shape — is copy-pasted 4 times. If the contract changes, 4 sites must be updated.

Solution

A paginate function that accepts a Drizzle query builder and page/limit params, returns the standard { data, pagination } shape.

Wins

  • Locality: pagination logic in one module
  • Leverage: one interface, 4+ call sites
  • Interface shrinks: caller passes query, gets result
  • 4 shallow implementations → 1 deep module

Extract the Order state machine into its own deep module

Strong in-process

features/orders/orders.service.ts (VALID_TRANSITIONS, canTransition, transitionStatus), new features/orders/state-machine.ts

Before — deep rules buried in shallow service

orders.service.ts

VALID_TRANSITIONS map
canTransition(from, to)
DB: getOrder(id, userId)
role authorization if/else chain
timestamp field map
DB: update() returning()

Deep business rules interleaved with shallow DB queries. Only testable through full HTTP stack.

After — deep module with clean interface

OrderStateMachine

transition(from, to, role) → { allowed, timestampField?, error? }

orders.service.ts (now shallow)

DB: getOrder(id, userId)
machine.transition(from, to, role)
DB: update() returning()

State machine testable without DB. Service becomes a thin adapter.

Problem

The order state machine — VALID_TRANSITIONS, transition validation, role authorization rules, timestamp field mapping — is embedded in orders.service.ts alongside raw Drizzle queries. The deep part is surrounded by shallow plumbing.

Solution

Extract the state machine into its own module with a single-function interface: transition(from, to, role) → result. The service becomes an adapter that calls the machine, then executes the DB update.

Wins

  • Locality: all transition rules in one module
  • Interface is the test surface — testable without DB
  • Leverage: state machine reused if webhooks arrive
  • Service shrinks to: fetch, decide, write

Deduplicate ownership verification across services

Worth exploring in-process

listings.service.ts (update, remove), orders.service.ts (getOrder, transitionStatus), new shared/guards.ts

Before — authorization scattered across service methods

listings.update() 
if (listing.sellerId !== sellerId)
  throw new ForbiddenError(...)
listings.remove() 
if (listing.sellerId !== sellerId)
  throw new ForbiddenError(...)
orders.getOrder() 
if (order.buyerId !== userId &&
    order.sellerId !== userId)
  throw new ForbiddenError(...)
orders.transitionStatus() 
if (newStatus === "paid" &&
    order.buyerId !== userId)
  throw new ForbiddenError(...)
// ... repeated per status

Same pattern, different conditions, no shared language.

After — guards consolidate authorization rules

ensureOwner

(resource, userId) → void | throws ForbiddenError

ensureParticipant

(order, userId) → void | throws ForbiddenError

update() → ensureOwner(listing, sellerId)
remove() → ensureOwner(listing, sellerId)
getOrder() → ensureParticipant(order, userId)

Authorization vocabulary in one module.

Problem

The pattern "check sellerId !== userId, throw ForbiddenError" repeats across listings and orders services. Authorization logic has no locality.

Solution

Extract ensureOwner(resource, userId) and ensureParticipant(order, userId) into a shared guards module. Services compose them: fetch, guard, act.

Wins

  • Locality: authorization rules in one module
  • Leverage: two guard functions serve all features
  • Consistent error messages and codes
  • Services read: "guard, then act"

Co-locate OpenAPI path definitions with features

Worth exploring in-process

shared/openapi.ts (460 lines → ~50), new features/*/openapi.ts

Before — monolith spec couples unrelated features

shared/openapi.ts (460 lines)
imports from auth.schemas
imports from listings.schemas
imports from orders.schemas
...
/api/v1/auth/* paths
/api/v1/listings/* paths
/api/v1/orders/* paths

Deletion test: removing the auth feature requires surgical extraction from this file.

After — each feature exports its own paths

features/auth/openapi.ts exports authPaths, authSchemas
features/listings/openapi.ts exports listingPaths, listingSchemas
features/orders/openapi.ts exports orderPaths, orderSchemas
shared/openapi.ts merges feature specs + shared schemas

Deletion test: delete features/auth/ → docs vanish with it. Matches the stated architecture.

Problem

The OpenAPI spec is a single 460-line file importing from all three feature domains. It contradicts the project's stated architecture ("features co-locate routes, schemas, and service logic"). Every new endpoint grows this file linearly.

Solution

Each feature exports its own path definitions and component schemas from features/X/openapi.ts. shared/openapi.ts becomes a thin merge point.

Wins

  • Locality: each feature's docs live with its code
  • Deletion test passes cleanly per feature
  • Parallel editing without merge conflicts
  • Matches existing feature co-location pattern

Split db connection management from schema re-export

db/index.ts, db/schema.ts, all feature service files

Before — two concerns, one module

db/index.ts has two interfaces: connection lifecycle and schema access.

After — one concern per module

                flowchart LR
                  A[auth.service] --> DB[db/index.ts]
                  A --> SC2[db/schema.ts]
                  B[listings.service] --> DB
                  B --> SC2
                  C[orders.service] --> DB
                  C --> SC2

Each module has one responsibility. Schema import path is direct.

Problem

db/index.ts both creates the pg Pool + Drizzle instance and re-exports the schema. Two interfaces: connection management and schema access. Every feature crosses this module for both concerns.

Solution

Import schema directly from db/schema.js where it's defined. db/index.ts stays as the connection module. 3 import path changes.

Wins

  • One responsibility per module
  • Schema import path is direct
  • Two interfaces instead of one merged interface

Top recommendation

Collapse try/catch into an async route wrapper

Smallest change with the widest impact. Touches every route file, eliminates ~25 lines of boilerplate, and the win is immediately visible in every route handler. The wrapper is 4 lines of code. It's the definition of depth: a large amount of behaviour (error forwarding for every route) behind a tiny interface (asyncHandler(fn)). Do this first — every subsequent refactor is cleaner because route handlers stop pretending to manage their own error flow.
