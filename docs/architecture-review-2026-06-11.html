Problem

Stripe interactions — payment intents, transfers, refunds — are scattered across orders.service.ts, complete-order.ts, and webhooks.service.ts. executeTransition is exported publicly but only called by external modules. complete-order.ts depends on orders.service.ts — a "ripped out" module that still calls back into its parent. The Order Service has 10 public exports with no clear interface.

Solution

Consolidate all Stripe interaction into a single Payments adapter behind one seam. Fold complete-order.ts back into the Order Service — the orchestration lives there, the Stripe calls are delegated. Make executeTransition private; expose a single transitionOrder that handles validation + persistence. The webhook module calls that one interface instead of composing transition() + executeTransition() manually.

Wins

  • Locality: all Stripe knowledge in one module
  • Leverage: one "execute payment" interface, 5 call sites
  • Interface shrinks: 10 exports → ~7, one internal seam removed
  • complete-order.ts disappears — tests for its orchestration move to Order Service tests
  • Webhook module no longer composes transition internals

2. Listing Reservation leaks across feature seams

After — reservations as first-class shared concern

flowchart LR
  listings["listings.service.ts"] -->|depends| reservation["reservation.ts\n(shared module)"]
  orders["orders.service.ts"] -->|depends| reservation
  reservation --> db["DB"]

  classDef deep fill:#0f172a,color:#fff,stroke:#0f172a
  class reservation deep

Problem

listings.service.ts imports resolveListingReservation from ../orders/expiry.js. The Listing module knows about Order expiry internals — it calls into the Order module to resolve a listing's effective status. This is a cross-feature leak: the Listing feature shouldn't know that a "reserved" listing gets its status from a pending Order's age.

Solution

Extract the Reservation concept into a shared module at src/shared/reservation.ts (or as a first-class domain module). Both Listings and Orders depend on it. The interface is one function: resolveListingStatus(listingId) → returns the effective status, handling stale reservation expiry internally. Alternatively, push the resolution up: the Listing route calls the Reservation module before passing the status to the Listing service.

Wins

  • Locality: reservation expiry logic in one place, not imported across features
  • Deletion test: if reservation logic is deleted, both features break — it earns its keep
  • Two adapters justify the seam: Listings and Orders both call it
  • Listings module no longer imports from Orders

3. Expiry module is shallow — 6 exports, overlapping concerns

src/features/orders/expiry.ts

After — deep module: 2 exports, more behaviour behind each

Interface

resolveListingStatus(id)

Implementation

expire + release (private)
DB-side now() staleness check
get pending order (private)
orchestration (private)

2 interface points, same implementation depth

Problem

The module exports 6 symbols. expireOrderAndReleaseListing and expireOrderAndReleaseListingIfStale differ only in a staleness check. expireIfStale is a thin status guard that delegates. getPendingOrderOnListing is a query helper used by resolveListingReservation. The interface nearly matches the implementation line-for-line — a shallow module by the glossary definition.

Solution

Collapse to 2 exports: expireIfStale(order) for the pay/cancel/webhook paths, and resolveListingStatus(listingId) for the order-creation and listing-access paths. Make the internal helpers private. The constant ORDER_EXPIRY_MINUTES stays exported (it's a config value consumed by other modules). The DB queries, staleness check, and listing release become private implementation details.

Wins

  • Interface shrinks: 6 → 2 exports, same behaviour behind them
  • Leverage: each call site only needs to know one function, not the internal choreography
  • Deletion test: the 4 thin wrappers disappear, complexity concentrates in the 2 remaining functions
  • Internal seams become truly internal — no caller can reach past the interface

Problem

features/payments/ has no routes, no feature surface. It's consumed infrastructure — a Stripe client singleton, amount conversion utilities, and error mapping. Its location under features/ misleads: it looks like a feature module that would have its own API endpoints, but it's a set of adapters consumed by four other modules. This is a naming / directory-structure issue that causes navigation friction.

Solution

Move features/payments/ to shared/payments/ (alongside shared/config.ts, shared/logger.ts). The Stripe client, amount utils, and error mapping are infrastructure — they belong with other infrastructure. If combined with Candidate #1, this module deepens into a single Stripe adapter behind one seam. The __tests__/ directory moves with it.

Wins

  • Navigation: one less entry in features/ that isn't a feature
  • Aligns with project structure: shared infrastructure in shared/
  • Sets up Candidate #1 — the Stripe adapter has a natural home
  • Low risk: pure file move, no interface change

5. guards.ts is a shallow module — two one-liners used once each

src/shared/guards.ts

Speculative

Before

ensureOwner(listing) → ForbiddenError if sellerId != userId
ensureParticipant(order) → ForbiddenError if neither buyer nor seller
Used once each (listings.service, orders.service)

After — inline or consolidate

Option A: Inline the checks. They're one line each.
Each caller already throws a typed error nearby.
Option B: Merge into errors.ts as static factory methods.
ensureOwner → resource ownership assertion as AppError factory.

Problem

ensureOwner is a 4-line function called exactly once in listings.service.ts. ensureParticipant is a 4-line function called exactly once in orders.service.ts. Both are thin wrappers around a condition + ForbiddenError throw. The module has its own test file. The deletion test: if you deleted guards.ts, the complexity reappears in two places — but it's one line in each. The module doesn't hide enough complexity to justify its existence.

Solution

Two reasonable paths. Option A: inline the checks — the condition + throw pattern is already idiomatic in the codebase (see payOrder, cancelOrder, refundOrder all do ad-hoc role checks). Option B: deepen by attaching guards to the domain types — e.g. a Listing method that asserts ownership, an Order method that asserts participation. This would concentrate authorization logic with the data it guards.

Wins

  • One fewer module to navigate for 8 lines of code
  • Inline pattern is consistent with existing ad-hoc checks
  • Guards test file shrinks or disappears

Top recommendation

Candidate 1: Order Service grab-bag

This is the highest-impact deepening. The Order Service is the most complex module in the codebase and it's leaking Stripe concerns across three files. Consolidating into one Payments adapter behind a clean seam would shrink the interface, concentrate all Stripe knowledge, and eliminate complete-order.ts as a separate file. Candidates 2 (Reservation leak) and 3 (Expiry depth) are natural follow-ups once Candidate 1 is done — they all touch the same call sites.
