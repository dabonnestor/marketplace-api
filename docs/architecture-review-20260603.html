Extract a Commission module

Strong local-substitutable
src/features/orders/orders.service.ts:47-51 src/features/orders/orders.schemas.ts:13

Before

subtotal = Number(price)

Fee logic untestable in isolation — entangled with DB + Stripe

After

calculateOrderBreakdown(subtotal, shippingCost)
→ platformFee: "10.00"
→ total: "105.00"
→ sellerPayout: "95.00"
Pure function · 0 deps · trivially testable
Fee edge cases (rounding, zero, negatives) tested at the seam

Problem

The 10% platform fee, rounding, and Order financial breakdown is inlined in createOrder. Testing edge cases (rounding, large amounts) requires exercising the full Order creation path with DB and Stripe.

Solution

Extract a pure Commission module: one function that takes a subtotal and shippingCost, returns platformFee, total, and sellerPayout as decimal strings. The PLATFORM_FEE_PERCENT constant moves in with it.

Wins

  • Locality: fee logic lives in one module
  • Interface shrinks to 2 inputs, 3 outputs
  • Tests: pure function, no harness needed
  • Leverage: one impl, used by createOrder + any future fee-aware path

Problem

The pattern "call transition(), build an updates object with timestamp fields, call db.update(orders)" appears in 5 places across two files. The webhook module leaks across the Order seam by reaching directly into the DB.

Solution

Move the "validate + build updates + persist" trio into transitionStatus (or a new executeTransition module that both orders.service and webhooks.service import). The webhook never touches schema.orders directly.

Wins

  • Locality: transition persistence logic in one module
  • Webhook module's interface shrinks: it only maps events → transitions
  • Deletion test: removing the 4 duplicate blocks concentrates behaviour
  • Idempotency guarantee enforced at the seam, not per call site
Caution: webhooks.service sometimes needs extra fields in the update (e.g. preDisputeStatus: null on dispute won). The consolidated module's interface must accept optional extra columns, or the webhook should set those before/after the transition call.

    classDef deep fill:#0f172a,stroke:#334155,color:#e2e8f0;

Transfer is an explicit step in the completion flow, not a surprise in a generic function

Problem

transitionStatus is named as a generic state transition, but it has a hardcoded branch: if (newStatus === "completed") that fetches the seller's Stripe account and creates a transfer. The transfer failure handling (502 error) also lives here, even though no other status transition can fail with a 502.

Solution

Create a completeOrder function (or inline in the route) that calls transitionStatus for the state change, then executes the Stripe transfer as a separate, explicit step. If the transfer fails, the order stays in delivered — same behaviour, clearer code.

Wins

  • Interface clarity: transitionStatus no longer has hidden Stripe dep
  • Locality: transfer logic + error handling in one place
  • Each module does one thing: transition state OR execute transfer
  • Tests: transfer failure tested without mocking the state machine

Centralise expiry enforcement

Worth exploring local-substitutable
src/features/orders/expiry.ts src/features/orders/orders.service.ts (4 call sites) src/features/listings/listings.service.ts:55-61

Before

createOrder: checks expiry before creating new order
payOrder: checks expiry before confirming payment
cancelOrder: checks expiry before cancelling
listings.getById: checks expiry before showing listing
Every caller duplicates the expiry guard. Forgetting one means stale reservations.

After

createOrder
payOrder
cancelOrder
listings.getById
resolveListingReservation(listingId)
Single guard. If expired → release. Returns current status.
One seam. Four callers. Impossible to forget the check.

Problem

The lazy-expiry pattern (isOrderExpired + expireOrderAndReleaseListing) is duplicated at 4 call sites. Adding a 5th operation that touches a reserved listing would require remembering to copy the pattern. The listings module imports from ../orders/expiry — a dependency direction that signals a missing seam.

Solution

Create resolveListingReservation(listingId): checks for a pending order, expires it if stale, returns the listing's effective status. Callers ask "is this listing available?" instead of independently deciding when to check expiry.

Wins

  • Locality: expiry policy lives in one module
  • Expiry window (30 min) changeable in one place
  • No cross-feature imports (listings ⇆ orders)
  • Interface: one function, one responsibility
Top recommendation

Extract a Commission module

Start here. It's the smallest, safest deepening with the highest testability payoff. A pure function with zero dependencies that takes two numbers and returns three decimal strings. No DB, no Stripe, no HTTP. The current code already works — this just gives the logic a name and a seam. Every other candidate touches DB or Stripe; this one is a clean win that establishes the pattern for the rest. Once the Commission module exists, extracting the PaymentIntent adapter (candidate 2) is the natural next step — both shrink the Order creation module from ~40 lines to ~15.
